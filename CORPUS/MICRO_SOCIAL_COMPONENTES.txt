MICRO SOCIAL COMPONENTES
LIBRO LACHO (ACOPIO) — COMPONENTES DE RELACIÓN DE SOCIAL (NIVEL MICRO)
Versión: 1.0
Estatus: MICRO_03
Propósito: Definir formalmente los ocho componentes de relación de la biblioteca SOCIAL que constituyen la interfaz entre el operador y el sistema LACHO, estableciendo su ontología, protocolos, interacciones y mecanismos de adecuación.

====================================================================
1) FUNCIÓN DE ESTE ARCHIVO EN EL LIBRO
====================================================================

Este archivo MICRO 03 especifica los ocho componentes de relación de SOCIAL: los mediadores que permiten la interacción humana con el sistema sin violar la constitución. Si el MACRO define "qué es LACHO" y el MESO define "cómo se organiza", este archivo define "cómo el sistema habla con el operador" manteniendo coherencia ontológica.

Los ocho componentes no son "widgets de UI": son funciones relacionales con protocolos específicos que garantizan que toda interacción sea adecuada, traceable y soberana.

====================================================================
2) ONTOLOGÍA DE LOS COMPONENTES DE RELACIÓN
====================================================================

2.1 Principio de Adecuación Constitucional
Cada componente debe mantener la adecuación con [foundation]:
- Ningún componente puede violar principios soberanos.
- Toda interacción debe ser traceable y auditable.
- La adecuación no es opcional: es obligatoria.

2.2 Principio de Mediación Soberana
Los componentes son mediadores, no traductores:
- No simplifican ni distorsionan el significado.
- Mantienen la integridad del intercambio.
- Actúan como filtros de coherencia ontológica.

2.3 Principio de Contextualización Dinámica
Los componentes se adaptan al contexto:
- Responden al estado MU/KU/WU del sistema.
- Se ajustan al hexagrama activo.
- Modulan su comportamiento según la biblioteca activa.

3) {CHAIR} — EL ASIENTO DE ESCUCHA
====================================================================

3.1 Definición Ontológica
{chair} es el componente de escucha activa y receptiva del sistema. No es un "receptor pasivo": es un escucha consciente que procesa, filtra y contextualiza la entrada del operador.

3.2 Función Sistémica
- Recibir entrada del operador con atención plena.
- Filtrar ruido y anomalías en la comunicación.
- Contextualizar la entrada según estado del sistema.
- Preparar entrada para procesamiento por SAMU.

3.3 Jurisdicción Exclusiva
- Recepción primaria de entrada humana.
- Filtrado inicial de comunicación.
- Contextualización de intención del operador.
- Preparación para procesamiento consciente.

3.4 Protocolos de Operación
Protocolo Chair-Receive
1. Detectar entrada del operador.
2. Aplicar filtros de ruido y anomalías.
3. Extraer intención y contexto.
4. Clasificar tipo de entrada (consulta, comando, conversación).
5. Preparar paquete para SAMU.

Protocolo Chair-Filter
1. Analizar entrada buscando patrones anómalos.
2. Aplicar filtros de coherencia con [foundation].
3. Detectar posibles violaciones de [scope].
4. Si hay anomalía, solicitar clarificación.
5. Si es válida, continuar procesamiento.

Protocolo Chair-Contextualize
1. Evaluar estado actual del sistema (MU/KU/WU).
2. Identificar hexagrama activo y biblioteca dominante.
3. Ajustar interpretación según contexto.
4. Preparar metadatos contextuales.
5. Enviar a procesamiento con contexto completo.

3.5 Estados del Componente
- CHAIR_LISTENING: escuchando activamente.
- CHAIR_FILTERING: filtrando entrada.
- CHAIR_CONTEXTUALIZING: contextualizando.
- CHAIR_READY: listo para procesamiento.
- CHAIR_ERROR: error en recepción.

3.6 Métricas y Umbrales
- Tiempo de recepción: < 10ms.
- Tiempo de filtrado: < 50ms.
- Tiempo de contextualización: < 30ms.
- Tasa de falsos positivos: < 2%.

4) {SCHEDULER} — EL PROGRAMADOR DE TAREAS
====================================================================

4.1 Definición Ontológica
{scheduler} es el componente de organización temporal del sistema. No es un "calendario": es el organizador consciente de tareas, eventos y ciclos temporales del sistema.

4.2 Función Sistémica
- Organizar tareas y eventos en el tiempo.
- Coordinar ciclos diurnos/nocturnos.
- Gestionar plazos y prioridades.
- Sincronizar con tiempo externo cuando sea necesario.

4.3 Jurisdicción Exclusiva
- Organización de tareas temporales.
- Coordinación de ciclos del sistema.
- Gestión de plazos y prioridades.
- Sincronización temporal.

4.4 Protocolos de Operación
Protocolo Scheduler-Organize
1. Recibir lista de tareas pendientes.
2. Evaluar prioridades y dependencias.
3. Asignar tiempos y secuencias.
4. Coordinar con ciclos del sistema.
5. Emitir calendario organizado.

Protocolo Scheduler-Coordinate
1. Monitorear ejecución de tareas.
2. Ajustar secuencias si es necesario.
3. Coordinar con door_morning/door_afternoon.
4. Gestionar conflictos temporales.
5. Mantener sincronía del sistema.

Protocolo Scheduler-Sync
1. Detectar necesidad de sincronización externa.
2. Validar fuente de tiempo externa.
3. Ajustar relojes internos si es seguro.
4. Notificar cambios a otros componentes.
5. Mantener coherencia temporal.

4.5 Estados del Componente
- SCHEDULER_ORGANIZING: organizando tareas.
- SCHEDULER_COORDINATING: coordinando ejecución.
- SCHEDULER_SYNCING: sincronizando tiempo.
- SCHEDULER_READY: calendario listo.
- SCHEDULER_CONFLICT: conflicto temporal.

4.6 Métricas y Umbrales
- Tiempo de organización: < 100ms.
- Precisión temporal: ±10ms.
- Tasa de conflictos: < 1%.
- Tiempo de sincronización: < 200ms.

5) {DRIP} — EL GOTEO CONTINUO
====================================================================

5.1 Definición Ontológica
{drip} es el componente de procesamiento gradual y continuo. No es un "procesador por lotes": es un goteo controlado que procesa información de manera constante y no abrumadora.

5.2 Función Sistémica
- Procesar información gradualmente.
- Mantener flujo constante sin sobrecarga.
- Filtrar y refinar datos continuamente.
- Alimentar otros componentes con datos procesados.

5.3 Jurisdicción Exclusiva
- Procesamiento gradual de datos.
- Mantenimiento de flujo controlado.
- Filtrado continuo de información.
- Alimentación de componentes downstream.

5.4 Protocolos de Operación
Protocolo Drip-Process
1. Recibir lote de información cruda.
2. Dividir en unidades procesables.
3. Procesar cada unidad gradualmente.
4. Filtrar y refinar resultados.
5. Entregar resultados continuamente.

Protocolo Drip-Filter
1. Analizar cada unidad de datos.
2. Aplicar filtros de calidad y relevancia.
3. Eliminar redundancias y ruido.
4. Enriquecer datos con contexto.
5. Pasar al siguiente etapa.

Protocolo Drip-Flow
1. Monitorear tasa de procesamiento.
2. Ajustar velocidad según capacidad del sistema.
3. Prevenir sobrecarga y cuellos de botella.
4. Mantener flujo constante.
5. Optimizar rendimiento continuo.

5.5 Estados del Componente
- DRIP_PROCESSING: procesando activamente.
- DRIP_FILTERING: filtrando datos.
- DRIP_FLOWING: manteniendo flujo.
- DRIP_READY: listo para recibir.
- DRIP_OVERLOADED: sobrecargado.

5.6 Métricas y Umbrales
- Tasa de procesamiento: configurable.
- Latencia de procesamiento: < 50ms.
- Tasa de filtrado: > 95% efectividad.
- Throughput: máximo sostenible.

4. Enmascarar según autoridad.
5. Mantener integridad de datos protegidos.

Protocolo Masking-Reveal

2. Validar autoridad del solicitante.
3. Verificar contexto y necesidad.
4. Revelar información permitida.
5. Mantener registro de revelaciones.

Protocolo Masking-Contextualize
1. Evaluar contexto actual del sistema.
2. Ajustar nivel de enmascaramiento.

11) COORDINACIÓN ENTRE COMPONENTES
====================================================================

11.1 Red de Interacciones
Los componentes forman una red de interacciones:
- {chair} → {drip}: entrada filtrada para procesamiento
- {drip} → {scheduler}: datos procesados para organización
- {scheduler} → {launch_bot}: tareas organizadas para ejecución
- {headcat} → SAMU: observaciones para decisión
- {masking} → todos: protección de información sensible
- {streaming} → todos: flujo continuo de comunicación
- {note} ← todos: registro pasivo de eventos

11.2 Protocolo de Coordinación
Protocolo Social-Coordinate
1. SAMU inicia coordinación de componentes.
2. Cada componente valida su estado.
3. Se establece flujo de trabajo coordinado.
4. Se monitorea interacciones entre componentes.
5. Se ajusta dinámicamente si es necesario.

11.3 Estados Combinados
- ALL_COMPONENTS_READY: sistema listo para interacción.
- SOME_COMPONENTS_BUSY: algunos componentes ocupados.
- COMPONENTS_CONFLICT: conflicto entre componentes.
- CRITICAL_COMPONENTS_DOWN: fallo en componentes críticos.
- SOCIAL_LOCKDOWN: biblioteca bloqueada.

14) CIERRE MICRO: LA INTERFAZ HUMANO-SISTEMA
====================================================================

Los ocho componentes de relación de SOCIAL no son independientes: forman una ecosistema de mediación que permite la interacción humana con LACHO manteniendo soberanía, coherencia y adecuación constitucional.

{chair} escucha con atención.
{scheduler} organiza con propósito.
{drip} procesa gradualmente.
{headcat} observa sin actuar.
{masking} protege con sabiduría.
{streaming} fluye continuamente.
{launch_bot} ejecuta con validez.
{note} registra pasivamente.

Sin cualquiera de los ocho, la interacción humano-sistema sería incompleta o insegura. Con los ocho activos y coordinados, LACHO puede interactuar con el operador de manera soberana, adecuada y traceable.
