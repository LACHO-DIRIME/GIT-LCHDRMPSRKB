## 🔄 BIBLIOTECA ACTIVITY: CICLOS, AGENCIA Y RELACIÓN ONTOLÓGICA (UF[H01]…UF[H64])

---

### 1.0 DEFINICIÓN

**ACTIVITY** es la biblioteca intermedia de LACHO: gobierna **ciclos de vida**, **transiciones**, **estados temporales**, y la **agencialidad** (cómo una intención se convierte en conducta sostenida sin violar [foundation], [scope] y [term]).

ACTIVITY no es un estado efímero: es la capa que define **qué estados son válidos** y **cuándo se cierran**.

---

### 1.1 JURISDICCIÓN (NO MEZCLAR CAPAS)

ACTIVITY opera como biblioteca relacional ontológica/agencial:

- **TRUST** define [foundation]/[scope]/[term].
- **SAMU @** audita y modula S scalar (dos fases: contar/escuchar).
- **GATE** orquesta el tiempo (latencia/entrada).
- **ACTIVITY** define ciclos y módulos UF[H##] (1/64) como gramática de transición.
- **WORK** ejecuta lo físico.
- **STACKING** cristaliza la memoria/forma.
- **SOCIAL** es interfaz declarativa.
- **CRYPTO** firma/identifica y protege.

Declaración:
```
[TRUST] → SAMU @ → GATE → ACTIVITY(UF[H01..H64]) → WORK/STACKING
```

---

### 1.2 CONDICIONES SOBERANAS (TRUST MICRO_01)

ACTIVITY solo puede operar si la tríada soberana está vigente:

- **[foundation]**: suelo técnico. Si está comprometido, se detiene todo.
- **[scope]**: perímetro de realidad. Delimita recursos y permisos.
- **[term]**: finitud obligatoria. Ningún ciclo ACTIVITY existe sin cierre.

**Precedencia:**
1. [foundation]
2. [scope] ajustado a [foundation]
3. [term] aplicado dentro de [scope]

**Declaración mínima de todo ciclo ACTIVITY:**
```
CYCLE_START → Foundation-Check → Scope-Definition → Term-Declaration
CYCLE_RUN   → SAMU @ monitor → Term-Activation → Term-Execution
CYCLE_END   → Stacking-Crystal + Crypto-Attestation + SOCIAL {note} + TOOLCHAIN
```

---

### 1.3 AUTORIDAD OPERATIVA (CRYPTO MICRO_02)

ACTIVITY no es autoridad: ACTIVITY es ciclo. La autoridad se expresa por asientos CRYPTO bajo auditoría SAMU:

- **(direction seat):** valida rumbo y propósito del ciclo.
- **(key seat):** valida identidad/permiso para entrar en [scope].
- **(ignition seat):** autoriza inicio de ejecución (y detiene lo no autorizado).
- **(shield seat):** defensa/contención ante amenazas.
- **(flow seat):** regula caudal/volumen de ejecución.
- **(link seat):** valida enlaces y cifrado de conexiones.

**Regla:** si el ciclo ACTIVITY produce artefacto ejecutable (local/cloud), debe quedar atestado por CRYPTO y sellado por SAMU @.

---

### 1.4 INTERFAZ RELACIONAL (SOCIAL MICRO_03)

ACTIVITY no conversa directo: ACTIVITY se expresa por componentes SOCIAL.

- **{chair}** recibe intención y prepara paquete para SAMU.
- **{scheduler}** ordena tiempo/ciclos.
- **{drip}** procesa gradual y continuo para no saturar.
- **{note}** registra memoria simple del ciclo.
- **{streaming}** distribuye cuando la salida está validada.
- **{masking}** protege y revela lo necesario según autoridad.
- **{headcat}** observa y reporta a SAMU sin ejecutar.

**Regla:** todo ciclo ACTIVITY debe dejar rastro mínimo en `{note}` y/o `{drip}` y quedar disponible para STACKING.

---

### 1.5 MEMORIA Y FORMA (STACKING MICRO_04)

STACKING es la capa que cristaliza la memoria/forma de los ciclos ACTIVITY:

- **{manifest}** registra el ciclo completo.
- **{policy}** define la política de ejecución.
- **{attestation}** proporciona la prueba de ejecución.

**Regla:** todo ciclo ACTIVITY debe quedar registrado en STACKING.

---

### 2.0 OBJETIVO NUEVO: BIBLIOTECAS RELACIONALES ONTOLÓGICAS AGENCIALES

ACTIVITY se extiende para construir **bibliotecas relacionales ontológicas** orientadas a agencia:

- **Relacional**: registra relaciones (entidad→entidad) y (estado→estado) como grafos auditables.
- **Ontológica**: define tipos, roles, restricciones y compatibilidades entre estados.
- **Agencial**: define políticas de acción (qué puede hacerse) y cierre (cuándo termina).

Esto permite crear *pequeños módulos* que:
- razonan localmente,
- actúan bajo [term],
- y dejan trazabilidad para RAG.

---

### 3.0 MODELOS PEQUEÑOS (LOCAL/CLOUD) BASADOS EN RAG LACHO

ACTIVITY incorpora un patrón de arquitectura para **pequeños modelos** (local/cloud) que operan como *operarios agenciales* bajo autoridad:

#### 3.1 Imagen Gigante vs Memoria Pequeña
- **Imagen Gigante**: OS local (recursos masivos, sensores, archivos).
- **Memoria Pequeña**: modelo de lenguaje pequeño (razón/Tr, 12GB RAM como referencia).

#### 3.2 Patrón de orquestación
```
ACTIVITY → define ciclo
RAG → aporta contexto (Behavioral/Cognitive)
SmallModel(local/cloud) → propone/elige acción dentro de [scope]
SAMU @ → audita deriva y modula S
STACKING → guarda manifest de ciclo + resultados
```

#### 3.3 Reglas para modelos pequeños
- **No sustituyen TRUST**.
- **No rompen [term]**: todo ciclo tiene cierre.
- **No emiten sin auditoría**: SAMU @ debe estar presente en el hilo.
- **No crean caja negra**: output debe quedar trazable (manifest + registro).

---

### 4.0 INTEGRACIÓN CON RAG (BEHAVIORAL / COGNITIVE)

ACTIVITY usa RAG como memoria operacional:

- **Behavioral RAG**: patrones de conducta y repetición.
- **Cognitive RAG**: grafo de conocimiento, síntesis, trazabilidad.

El objetivo de ACTIVITY es que todo ciclo agencial produzca:
- registro de patrón,
- justificación (Tr),
- y cierre por [term].


### 8.0 PROTOCOLOS PARA BIBLIOTECAS AGENCIALES (RELACIONES)

#### 8.1 Protocolo Activity-Relate (ontológico)
1. Declarar entidad/estado (UF + contexto).
2. Declarar relación (causa, dependencia, transición, bloqueo).
3. Validar coherencia (Ψ) y límites ([scope]/[term]).
4. Registrar manifest (qué, por qué, cuándo).
5. Cristalizar en STACKING.

#### 8.2 Protocolo Activity-Agent (agencial)
1. Seleccionar UF slot según estado MU/KU/WU.
2. Recuperar contexto por RAG (H48 si aplica).
3. Proponer acción por modelo pequeño (local/cloud).
4. Auditar por SAMU @ (S scalar + reglas).
5. Ejecutar en WORK si corresponde.
6. Registrar y cerrar por [term].

#### 8.3 Protocolo Activity-MiniModel (local/cloud)
- Local: preferido por default (soberanía).
- Cloud: permitido si `policy` lo habilita dentro de [scope].
- En ambos: output trazable y cerrable por term.

---

### 9.0 RELACIÓN CON SOCIAL Y STACKING (SUPERFICIAL → PROFUNDA)

- Superficial:
  - SOCIAL declara (pedido).
  - ACTIVITY ejecuta ciclo.
  - STACKING registra.

- Profunda:
  - ACTIVITY produce ontología relacional: grafos de estado y agencia.
  - STACKING cristaliza “forma” (pilares) del ciclo.
  - SAMU @ preserva integridad temporal.

---

### 10.0 CIERRE

ACTIVITY es el lugar donde LACHO escala sin romper: define UF[H01..H64] como alfabeto de ciclos, habilita bibliotecas relacionales ontológicas/agenciales, y permite operar con pequeños modelos (local/cloud) bajo RAG, con auditoría SAMU @, límites TRUST y cristalización STACKING.
